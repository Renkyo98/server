# 알려진 문제 (미해결, 추후 처리 예정)

## ✅ 해결됨 — item.c 필드 정렬 버그 (코드 버그, 수정 완료)

**증상**: 부팅 시 `[ITEM data Error] tbl=95:itemcolor` 등 다수의 아이템 파싱 오류.

**원인**: `item.c`의 아이템 읽기 루프에서, 공격/방어/순발력/HP/MP/행운/매력/회피/독/마비/수면/석화/취함/혼란/크리티컬
(총 15개) "랜덤값" 필드를 처리할 때, `readpos`(현재 읽는 필드 위치)를 추가로 1칸 증가시키는 코드가
`if( strlen(token) != 0 )` 안에 있었음. 즉 **해당 필드가 비어있는 아이템(장비가 아닌 소모품/재화/이벤트
아이템 등)은 이 추가 증가가 안 일어나서, 그 뒤 모든 필드 위치가 하나씩 밀리는 버그**였음.

**수정**: 이 증가 로직을 `if(strlen(token)!=0)` 블록 밖으로 빼서, 필드가 비어있어도 항상 실행되도록 수정.
(`item/item.c` 약 1236~1262번째 줄)

**추가로 발견해서 같이 고친 데이터 문제**:
- `itemset6.txt` 1~2번째 줄: 이름도 없이 반토막난 고장 데이터 → 삭제
- `itemset6.txt`의 21개 줄: 표준(95개) 대비 필드가 1개씩 부족 → 트레일링 콤마 추가로 보정
- `itemset6.txt`의 구간 제목 줄 3개(#없이 방치): `#` 주석 처리

**결과**: `[ITEM data Error]` 완전히 0건.

## 2. enemy.txt가 참조하는 enemybase.txt ID 243개 누락 (데이터 누락, 추후 처리)

**증상**: 부팅 로그에 `enemy.c:562 파일 문법 오류:./data/enemy.txt N번째 줄`이 수백 건 발생.

**원인**: `enemy.txt`(몬스터 스폰 정보)의 특정 행들이 참조하는 `enemybase.txt`(몬스터 스탯) ID가
실제로 존재하지 않음. **확인 결과 이건 최근 작업(도감 재배치 등)으로 생긴 문제가 아니라, 원본
(사용자가 처음 업로드한, 손대기 전) 데이터부터 이미 존재하던 누락**임.

**성격**: 코드 버그 아님 — 순수 데이터 누락. "외부 데이터 나중에 처리" 항목과 동일 카테고리.

**규모**: enemy.txt가 참조하지만 enemybase.txt에 없는 고유 ID **243개**.

**다음에 할 일**: 외부 데이터(펫/몬스터) 보강 작업 시, 이 243개 ID 목록을 기준으로 DATA.zip 등에서
찾아서 채우면 됨.

## 3. 부팅 시 "BOT failed" / ITEM_CHECKITEMTABLE(-1) / MAP_addNewObj 오류 (데이터 누락)

**증상**: `gmsv64` 부팅 중 `LoadAllbluesLUA("mylua")` 단계(main.c "초기화 LUA 스크립트..." 직후)에서
아래 오류들이 한 번에 몰아서 발생 (약 65건):

```
********[No Enemy Id]BOT failed.
ITEM_CHECKITEMTABLE() number:-1 ITEM_tblen:140001 !!
Can't Get ItemTable for:-1! - itembase.c:265
Can't makeItem for itemid:-1!!
object.c:69 MAP_addNewObj error
```

**원인**: `mylua/` 하위 어떤 스크립트가 boot 시점(top-level 코드)에서 `char.CreateBot(...)`류의
함수(내부적으로 `mylua/charbase.c`의 봇 생성 로직, ~3078번째 줄 `[No Enemy Id]BOT failed`)를
호출하는데, 이때 넘기는 `enemy.txt`의 ID가 현재 우리 데이터셋에 없어서 실패. 이게 연쇄적으로
아이템/오브젝트 생성 실패로 이어짐.

**코드 버그 아님** — 데이터 누락 문제. 나중에 외부 데이터(펫/아이템/enemy.txt) 보강 작업 때
같이 해결 예정. 위 2번 항목(enemybase ID 243개 누락)과 연관되어 있을 가능성 높음.
